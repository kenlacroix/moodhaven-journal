## What Changed

### 1. `src-tauri/src/commands/data_management.rs` (ac53d88)

**Import version allowlist** — `ALLOWED_IMPORT_VERSIONS` changed from:
```rust
&["1.0", "1.1", "1.2", "1.3"]
```
to:
```rust
&["1.0", "1.1", "1.1.0", "1.2", "1.3"]
```
`export_data` produces `"version": "1.1.0"`. The old list did not contain this string, making every backup made with the current app impossible to import.

**Windows path traversal** — `write_text_file` now has `#[cfg(target_os = "windows")]` blocked prefix guards for `C:\Windows\`, `C:\Program Files`, `C:\ProgramData`, and `C:\Users\All Users`. Previously only Unix paths (`/etc/`, `/usr/`) were blocked.

### 2. `src/lib/services/webdavService.ts` (bc43c00)

`validateWebDAVUrl` now calls `console.warn` when a `http://` URL is configured. No functional change — `http://` still works. The warning informs developers and surfaced logs that plaintext HTTP was deliberately allowed.

### 3. `src-tauri/src/commands/two_factor.rs` (78661f6)

`verify_totp_code` (the 2FA setup verification command) now shares the `PasswordRateLimiter` budget. The core verification logic was extracted into `verify_totp_code_inner` so `enable_totp` can call it without re-entering the rate limiter (enable_totp is already gated by a successful verify).

`gen/schemas/acl-manifests.json` was auto-regenerated by `cargo check` because the command signature changed (added `rate_limiter` Tauri state parameter). This is expected.

## What's Left (Not Applied)

### Binary restore frames unencrypted (Finding 3 — MEDIUM)
**Location:** `src-tauri/src/commands/peer_sync_engine/connection.rs:46-64`  
**Issue:** `write_binary_frame` / `read_binary_frame` are intentionally unencrypted. The peer sync engine comment says "already sent after an encrypted envelope confirming chunk metadata." But the raw SQLite DB bytes during `peer_full_restore` are not wrapped in the AES-GCM transport layer.  
**Why not applied:** Fixing this requires a breaking protocol change. Both sides must be on the same protocol version simultaneously, or version negotiation must be added to the HELLO exchange.  
**Proposal:** Add a protocol version field to HELLO. If both peers support v3, use `write_frame` (encrypted) for binary chunks. Fall back to `write_binary_frame` with peers on v1/v2. Implement in a dedicated PR targeting the next protocol version bump.

### Restore checksum fail-open (Finding 4 — MEDIUM)  
**Location:** `src-tauri/src/commands/peer_sync_engine/mod.rs:1743-1748`  
**Issue:** When `moodhaven_restore.pending.sha256` is absent, `peer_apply_and_restart` logs a warning and proceeds.  
**Why not applied:** This is backward-compat behavior for builds older than the checksum feature. All current builds write the `.sha256` file. The fail-open path is intentional and documented.  
**Future action:** After 2-3 release cycles, change the else branch to `return Err("...")`. Add a migration note in CHANGELOG.

## Adjacent Issues (Logged, Not Fixed)

These were observed during the audit but are out of scope for a security hardening pass:

1. **`peer_key.bin` stored unencrypted at rest** — The Ed25519 private key for peer sync is stored as raw bytes at `{app_data}/peer_key.bin` with `0600` permissions. It is never transmitted and not a crypto key for data encryption. However, if an attacker gets the DB file and this file, they can impersonate the device to trusted peers. Encrypting it with the user's password (like the TOTP secret) would harden this. Tracked as T10 in the pentest plan.

2. **WebDAV `username` not in `SENSITIVE_PATHS`** — `webdav.password` is AES-256-GCM encrypted before storage. `webdav.username` is stored plaintext in the `app_settings` blob. Low risk (username not a credential by itself) but inconsistent with the `password` treatment.

3. **`oura_sync_today` and `oura_backfill` receive decrypted PAT via IPC** — The Oura PAT is stored encrypted via `secureStorage`. Before calling these Rust commands, the frontend decrypts the PAT and passes it as a plain string in the IPC call. Within the Tauri process this is the same trust boundary, so this is not a vulnerability — but it means the PAT is briefly in JS memory as a plain string, which is expected and unavoidable.

4. **Ollama uses `http://localhost`** — `aiService.ts` hardcodes `http://localhost:11434` as the Ollama default. This is intentional (local-only LLM), but the `http://` scheme is visually inconsistent with the WebDAV warning. Not a security issue.

## Assumptions Made

1. **The threat model is desktop-local**: An attacker who has filesystem-level access to the running user's account has many more attack vectors than any of these findings provide. All findings are evaluated in the context of a remote/WebView attacker, not a local user.

2. **`binaries/` symlink**: Added a symlink `src-tauri/binaries -> /home/ken/Documents/GitHub/MoodBloom-Tauri/src-tauri/binaries` to allow `cargo check` in the worktree. This symlink is untracked (it's in the untracked file list, not staged). The worktree does not contain the binary at HEAD — this was a local dev workaround.

3. **`gen/schemas/acl-manifests.json` auto-update**: The TOTP fix changed the `verify_totp_code` command signature. Tauri's build system regenerates the ACL schema on `cargo check`. This auto-generated file is correctly committed alongside the code change.

4. **No crypto contract changes**: Every fix in this PR is at the application or input-validation layer. No changes touch: PBKDF2 parameters, AES-GCM IV/nonce generation, key derivation, salt lengths, or the EncryptedContent wire format. The encryption contract is unchanged.

5. **The Argon2 default parameters for hardware key** (`Argon2::default()` = m_cost 4096 KiB, t_cost 3, p_cost 1) are the argon2 crate's built-in defaults. For the hardware key use case (deriving a key from an Ed25519 assertion signature, not a user password), these are adequate. The signature is already a high-entropy input, so memory-hardness adds defense-in-depth but isn't critical.
