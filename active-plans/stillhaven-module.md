# Design: StillHaven — Somatic Companion Module for MoodHaven

Generated by /office-hours on 2026-05-10
Branch: main
Repo: kenlacroix/stillhaven (planning) + kenlacroix/moodhaven-journal (implementation)
Status: APPROVED
Mode: Builder

## Problem Statement

The user (40yo somatic practitioner, no clinical training) wants a personal wellness tool that closes a specific loop he runs many times per week:

> Notice activation (anxiety, fake-danger narrative, authority trigger, post-hike processing window) → check in (1–10 + manual HRV) → bilateral session in an immersive environment → check out (1–10) → land in a journal entry that's already pre-filled with what just happened.

He already ships a journaling app (MoodHaven, v0.9.4, Tauri+SQLite+React). StillHaven was originally scoped as an independent companion PWA sharing IndexedDB with MoodHaven. That premise turned out to be wrong: MoodHaven is a Tauri-first app deployed at `journal.moodhaven.app`, with web-mode IndexedDB as a fallback, AES-256-GCM encryption, Oura already integrated, P2P + WebDAV sync, 101 Tauri commands. Building StillHaven as a separate codebase would duplicate that infrastructure twice.

**Reframed:** StillHaven is a *module* inside the MoodHaven codebase. One Tauri binary, one web build, one storage layer, two product surfaces.

## What Makes This Cool

- **The underwater scene as the bilateral surface itself.** Not "play tones over a video." The light rays *are* the bilateral cues — alternating left and right at the user-set tempo. Fish swim cross-stream as ambient depth. A depth gauge doubles as the activation meter (deeper = more regulated). Breathwork timer is the bubble cadence. Personal: this references a childhood school trip to an aquarium and a scuba certification milestone.
- **One processing loop, end to end.** Most somatic apps are protocol libraries. This one closes the loop into a journal entry automatically — the data path is the product.
- **Wellness framing as engineering discipline.** No clinical language anywhere in the bundle. Lint rule enforces it. "EMDR" is a banned substring. "BLS" / "bilateral stim" / "activation" / "regulation" are fine. Keeps the tool legally safe and culturally honest.
- **Renderer abstraction designed for WebXR**, but day-one ship is a 2D canvas. The engine emits bilateral events; renderers subscribe — meaning a future WebXR renderer is a *new file*, not an engine rewrite. The "no rebuild" claim is unverified until a hello-world WebXR renderer exists; treat as design intent, not guarantee.

## Constraints

- **Solo developer, vibe-coding cadence.** Build first, refine after.
- **Privacy absolute.** No cloud round-trip. Activation samples + session logs stay local, encrypted at rest the same way journal entries are.
- **No clinical claims.** Hard rule, code-enforced.
- **Inherits MoodHaven's stack:** Tauri v2 / Rust / SQLite (rusqlite) / React 18 / TypeScript strict / Tailwind / Zustand stores / TipTap editor / Vitest. No new top-level dependencies unless justified.
- **Existing schema:** mood is 1–5; activation is 1–10. Different scales — keep them in different tables. Don't conflate.
- **Audio must work without granted device permissions.** Web Audio API only — no microphone, no `getUserMedia`.

## Premises

Revised after reading MoodHaven's actual structure. All five must hold for this design.

1. **Bilateral engine and renderer are separate concerns.** Engine emits typed events on a clock; renderers (audio, visual, haptic, future WebXR) subscribe. Adding a forest scene later should not touch the engine.
2. **One codebase, one storage layer.** StillHaven lives at `src/modules/stillhaven/` inside `moodhaven-journal`. SQLite schema gets new tables; IndexedDB browser backend gets new object stores. No cross-origin / cross-app sharing problem to solve.
3. **Wellness language is a hard rule, code-enforced.** ESLint rule + Rust clippy lint scan for banned strings (`EMDR`, `trauma`, `treatment`, `patient`, `diagnose`, etc.) and fail the build.
4. **First deliverable is the core loop end-to-end.** Audio bilateral + 2D underwater (canvas) + check-in/out + journal handoff. Every other feature (forest/sky envs, WebXR, haptics, voice memo, Oura auto-pull, breathwork timer, spatial audio, Tauri-side haptics) is sequenced *after* one clean run of the loop.
5. **Sessions are append-only event records.** Activation samples and journal entries reference them by `session_id`. Sessions are never edited; corrections are new annotations on the same `session_id`.

## Approaches Considered

### Approach A: Module inside MoodHaven (chosen)

- **Where:** `src/modules/stillhaven/` (frontend) + `src-tauri/src/commands/still.rs` (Rust commands) + extensions to `src-tauri/src/db/schema.sql`
- **Routes:** `journal.moodhaven.app/still` (web) + `/still/*` inside the Tauri window
- **Storage:** new SQLite tables (`still_sessions`, `still_activation_samples`); new IndexedDB object stores for web mode; `journal_entries` gets a nullable `session_id` column
- **Handoff:** in-process Zustand action `journalStore.draftFromSession(session)` that pre-fills the existing TipTap composer
- **Effort:** S — most of the surface area is already built (auth, encryption, Oura, sync, store patterns). Net new is the bilateral engine, the underwater renderer, the protocol UI, and the schema additions.
- **Risk:** Low — no architectural novelty; follows existing patterns.
- **Reuses:** TipTap composer, Zustand store conventions, browser-invoke shim, Tauri command pattern, encryption layer, Oura context (`useOuraContext.ts`), settings system, watch-companion conventions.

### Approach B: Two Tauri apps, shared `haven-core` Rust crate

- **Effort:** L — needs monorepo refactor, workspace Cargo setup, splitting MoodHaven's existing modules into `core` vs `app`.
- **Risk:** Med — existing crypto/Oura/sync code wasn't designed to be shared; refactor introduces regression risk to the shipped journal app.
- **Reuses:** the *idea* of MoodHaven's architecture; not the actual code, until extracted.
- **When this would be right:** if StillHaven needs to ship as a separately-installable binary that someone could run *without* the journal app. (User's intent doesn't require that.)

### Approach C: Fully separate codebases, sync via export/import

- **Effort:** XL — duplicate encryption, Oura, sync, settings, 2FA infrastructure.
- **Risk:** High — encryption mistakes become twice as likely; key management across two apps is a real failure mode.
- **Reuses:** nothing concrete — only design patterns.
- **When this would be right:** if the two apps must be developed by different teams with no shared release cadence. (User is solo.)

## Recommended Approach

**Option A.** Sequenced as follows.

### Phase 0 — Module skeleton + lint rules (1 session)

Scoped to purely structural changes. No user-facing surface ships in this phase.

- Create `src/modules/stillhaven/` with `index.tsx` (entry view stub), `engine/`, `environments/`, `components/`, `types.ts`.
- Extend the existing `ViewType` union in `src/components/layout/Sidebar.tsx` with `'still'`. Wire it into `App.tsx`'s `currentView` switch — but **render nothing user-visible** in Phase 0; the entry view returns `null` until Phase 4 fills it in. The `'stillSessions'` view is added in Phase 6.
- **Sidebar nav entry, BottomTabBar entry, and `Ctrl+Shift+S` keyboard shortcut are NOT shipped in Phase 0.** They ship in Phase 4 alongside the check-in UI they navigate to. (Decided D4: shipping nav to a dead view is worse than waiting.)
- **Feature flag (build-time):** read `import.meta.env.VITE_FEATURE_STILL` in App.tsx; when falsy, the `'still'` ViewType case in the switch renders `null` and the Sidebar nav entry (added in Phase 4) is omitted from the menu list. Dev builds set `VITE_FEATURE_STILL=true` in `.env.local`; production builds keep it unset until MVP is done. No runtime toggle, no SQLite, no settings UI. (Decided D5.)
- **Single source-of-truth banned-words file:** `wellness-language-banned.txt` at the repo root, one entry per line. Each line: `<word-or-regex>` plus an optional `regex:` prefix. Both linters read this file. (Decided D6.)
- **Lint rule (TS):** new local ESLint plugin at `eslint-plugins/no-clinical-language/index.cjs` (CommonJS), referenced via `--rulesdir ./eslint-plugins/no-clinical-language` in the lint script. - **Lint rule (Rust):** `build.rs` pre-build script in `src-tauri/` that walks `src/`, reads the same `wellness-language-banned.txt`, and fails the build if any banned word appears outside an `// allow-clinical:` annotation. ~50 lines including the file-loader. Runs every `cargo check`.
- Both lint rules fail CI.

### Phase 1 — Schema (1 session)

**Critical:** MoodHaven runs `schema.sql` via `conn.execute_batch(include_str!("schema.sql"))` on every startup (`db/mod.rs:226`). Runtime ALTERs live as separate `let _ = conn.execute(...)` calls in `Database::new()` so duplicate-column errors are silently swallowed (`db/mod.rs:231–301`). The schema work splits into two parts.

**Part A — append `CREATE TABLE IF NOT EXISTS` blocks to `src-tauri/src/db/schema.sql`:**

```sql
-- StillHaven: somatic session event log (append-only)
CREATE TABLE IF NOT EXISTS still_sessions (
    id TEXT PRIMARY KEY,                     -- UUID
    protocol TEXT NOT NULL,                  -- see protocol enum (constrained at app layer)
    environment TEXT NOT NULL DEFAULT 'underwater',
    bilateral_mode TEXT NOT NULL,            -- 'audio' for MVP; others reserved
    duration_seconds INTEGER NOT NULL,
    started_at TEXT NOT NULL,
    completed_at TEXT,                       -- NULL if abandoned
    abandoned_at TEXT,                       -- set when the user closes mid-session
    created_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS still_activation_samples (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    session_id TEXT NOT NULL REFERENCES still_sessions(id) ON DELETE CASCADE,
    phase TEXT NOT NULL CHECK (phase IN ('pre', 'post')),
    activation INTEGER NOT NULL CHECK (activation >= 1 AND activation <= 10),
    hrv_manual INTEGER,
    hrv_source TEXT,                         -- 'manual' | 'oura' | 'fitbit' | NULL
    note TEXT,
    sampled_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_still_sessions_started ON still_sessions(started_at DESC);
CREATE INDEX IF NOT EXISTS idx_still_sessions_protocol ON still_sessions(protocol);
CREATE INDEX IF NOT EXISTS idx_still_samples_session ON still_activation_samples(session_id);
CREATE INDEX IF NOT EXISTS idx_entries_session ON journal_entries(session_id);
```

(MVP's protocol enum is the *runtime-constrained* set of two: `general_activation` and `fake_danger`. The other four protocols ship in a later phase; the column stays open-text in the DB.)

**Part B — add the `journal_entries.session_id` column as a runtime ALTER in `db/mod.rs`**, immediately after the existing privacy_mode/location_weather block (~line 232) and BEFORE the existing `book_id` ALTER which references journal_entries (so column ordering is consistent with the existing pattern):

```rust
let _ = conn.execute(
    "ALTER TABLE journal_entries ADD COLUMN session_id TEXT",
    [],
);
```

The FK is enforced at the app layer (Rust commands check session existence before linking) — SQLite ALTER TABLE doesn't add FK constraints to existing tables anyway. Order matters: the still tables are created by `execute_batch(schema.sql)` in step 1, then the journal_entries ALTER runs in step 2 — both happen inside `Database::new()` with the still tables guaranteed to exist before any code references them.

**Part C — sync_log integration. Sessions ARE synced (revised).** MoodHaven has a `sync_log` table (`db/mod.rs:376`) with triggers on existing tables for incremental P2P/WebDAV sync. The earlier draft scoped sessions OUT of sync to honor "privacy absolute" — but `journal_entries` IS synced, and the new `journal_entries.session_id` column would replicate to device B even if `still_sessions` did not. Result: dangling FKs on every remote device. That's worse than syncing the sessions.

Resolution: add insert/update/delete triggers for `still_sessions` and `still_activation_samples` that write to `sync_log`, mirroring the existing pattern at `db/mod.rs:394–436` for journal_entries. This keeps the data model consistent: a journal entry that references a session arrives on device B together with the session row it points to. "Privacy absolute" is preserved by the existing sync model (encrypted-at-rest payloads, user-controlled WebDAV endpoint, P2P pairing). HRV and activation are no more sensitive than the journal text itself, which already syncs.

```sql
CREATE TRIGGER IF NOT EXISTS still_sessions_sync_insert AFTER INSERT ON still_sessions
BEGIN
    INSERT INTO sync_log (table_name, row_id, op, synced) VALUES ('still_sessions', NEW.id, 'insert', 0);
END;
-- Mirror update/delete triggers + same trio for still_activation_samples (cast row_id to text).
```

**Part D — IndexedDB additions in `src/lib/backend/browser.ts`:**

- New object stores: `still_sessions` (keyPath: `id`), `still_activation_samples` (keyPath: `id`, autoIncrement: true)
- New index on `journal_entries`: `by-session-id` over the `session_id` field
- Bump DB version; write `onupgradeneeded` migration that idempotently adds the stores and the index. Existing `journal_entries` records survive untouched (no data rewrite).

**Part E — Rust commands in `src-tauri/src/commands/still.rs`.** All commands follow the existing `journal.rs` pattern: take `lock: State<'_, AppLockState>` and call `require_unlocked(&lock)?` first. Naming follows the `oura.rs` convention (`domain_verb_noun`) — chosen because StillHaven is a self-contained module like Oura, not a core type like journal. Rationale documented in the commands' module doc comment.

```rust
#[tauri::command]
pub async fn still_create_session(
    db: State<'_, Database>,
    lock: State<'_, AppLockState>,
    protocol: String,
    environment: String,
    bilateral_mode: String,
    duration_seconds: i64,
) -> Result<String, String> { require_unlocked(&lock)?; /* ... */ }
```

Same shape for: `still_record_activation`, `still_complete_session`, `still_abandon_session`, `still_list_sessions`, `still_get_session_with_samples`. Each takes `lock` + `db` and gates on `require_unlocked`. **Browser-invoke shim** (`src/lib/backend/browser-invoke.ts`) implements every command against IndexedDB; web mode is fully functional without Tauri.

Register all commands in `src-tauri/src/lib.rs` (the existing `invoke_handler!` macro list).

### Phase 2 — Bilateral audio engine (1–2 sessions)

**Ownership:** the engine is a singleton instantiated lazily on first session start, owned by the StillHaven Zustand store. The store exposes `start(config)`, `stop()`, and a tick-event subscription. React components subscribe via a `useBilateralEngine()` hook that returns the store slice. When the StillHaven view unmounts mid-session, the engine continues running (so the user can briefly check elsewhere); when the user explicitly ends or abandons the session, `stop()` is called and the AudioContext is suspended.

**AudioContext lifecycle (decided D3, critical for browser autoplay policy and backgrounding):**

1. **Creation:** AudioContext is created lazily on the first `engine.start()` call, NOT at module load. This avoids ever creating a suspended context that never resumes.
2. **Gesture requirement:** `engine.start()` MUST be invoked synchronously inside a user-gesture handler (the "Start session" button's onClick). Implementation: the handler calls `engine.start()` directly — no `await` between the click and `AudioContext.resume()`. The session row is created via `invoke('still_create_session', ...)` AFTER `engine.start()` returns, not before. Safari and Chrome both enforce this; getting the order wrong = silent failure.
3. **Visibility change handler:** the engine installs a `document.visibilitychange` listener on `start()`. On `hidden`, the lookahead scheduler pauses but the AudioContext is NOT suspended explicitly (some browsers auto-suspend, some don't — let the browser decide). On `visible`, the scheduler resumes IF the AudioContext is still running; if it's suspended, the engine emits a `paused` event consumed by the session UI.
4. **AudioContext statechange handler:** subscribe to `audioContext.statechange`. When state goes to `'suspended'` or `'interrupted'` mid-session, emit a `paused` event. UI displays a "Paused — tap to resume" overlay with a single button that calls `engine.resume()` synchronously. `engine.resume()` calls `audioContext.resume()`, restarts the scheduler, and the session continues from where it was (session duration tracks elapsed scheduled time, not wall clock).
5. **Cleanup:** `engine.stop()` removes both listeners, suspends the AudioContext, and clears the scheduler interval.

This adds ~80 lines to the engine but covers the realistic phone-switching-to-Oura-app scenario which IS this user's primary use case.

`src/modules/stillhaven/engine/bilateralEngine.ts`. Core API:

```typescript
export type Side = 'L' | 'R';
export interface BilateralEvent { side: Side; t: number; }
export interface EngineConfig {
  speedHz: number;        // 0.5–2.0; default 1.0 (1 left + 1 right per second)
  toneHz: number;         // default 250
  volumeDb: number;       // default -18
  envelopeMs: number;     // default 30 — short ramp prevents clicks
}

export class BilateralEngine {
  constructor(config: EngineConfig);
  start(): void;
  stop(): void;
  onTick(handler: (e: BilateralEvent) => void): () => void;
  setSpeed(hz: number): void;
}
```

Implementation skeleton (Web Audio API, sample-accurate):

- One `AudioContext`. One `OscillatorNode` (sine, **default 200Hz** per D12/A). One `BiquadFilterNode` (low-pass at 800Hz) immediately after the oscillator — softens the harmonic edge so the tone reads as warm pulse, not clinical beep. One `GainNode` for envelope. One `StereoPannerNode` for hard-left / hard-right. Master `GainNode` for volume.
- Use a **lookahead scheduler pattern** (Chris Wilson, "A Tale of Two Clocks"): a `setInterval(25ms)` pre-schedules the next ~100ms of pan + gain changes via `setValueAtTime`. UI thread jitter doesn't affect timing.
- Each tick: schedule pan to one side, schedule gain ramp-up over `envelopeMs`, sustain for `(1/(2*speedHz)) - 2*envelopeMs/1000` seconds, ramp down, then alternate side on next tick.
- `onTick` fires on the UI thread (rAF-aligned) for renderers to subscribe — *visual* renderers use this to flash light rays in sync, but they don't drive the audio clock.

Why this shape: same engine instance feeds audio output, drives the underwater scene's left-right light rays, and (later) triggers haptic pulses via Tauri's notification plugin or a Web Vibration API call. Renderers are passive subscribers.

### Phase 3 — Underwater renderer, 2D canvas (1 session)

`src/modules/stillhaven/environments/underwater/Underwater2D.tsx`. A `<canvas>` full-bleed, no UI chrome visible. RequestAnimationFrame loop. Subscribes to `engine.onTick`.

**Approved visual direction (D9, variant C):** pure environment, no human figures, no statues, no figurative content. The user IS the diver in their imagination — third-person figures break first-person immersion. Reference mockup: `~/.gstack/projects/kenlacroix-stillhaven/designs/underwater-scene-20260518/variant-C.png`.

**Concrete palette + composition:**
- Background gradient: vertical, `#0a3a5a` (deep cobalt) at the top → `#1a6478` (teal) → `#2a8499` (lighter teal) toward the bottom third. Subtle textured sand suggested at very bottom edge.
- Bilateral light rays: warm gold `#f4d27a` at full brightness, soft falloff to transparent over ~30% of viewport height. Origin: upper-left and upper-right corners, angled at ~20° from vertical. Width: 8% of viewport. Engine emits ticks → rays alternate left/right at user's set speed; opacity ramps via the same envelope as the audio (~30ms attack/release).
- Ambient fish: 3–5 silhouettes in mid-distance, dark `#0a3a5a` against the gradient, slowly drifting left-to-right at ~10px/sec (independent of bilateral cadence). Spawn at right edge, exit at left edge, respawn after random 8–20s pause.
- Caustic particles: 30–50 small (~2–4px) translucent white circles, slow vertical drift with subtle horizontal sway. Pre-allocated array, no per-frame allocation. Re-pool on exit.
- Depth gauge: NOT visible during session (per D6/A). Only appears at check-out.
- No timer, no end button, no speed slider during session — controls appear on tap-to-reveal (per D5/A).

**Tap-to-reveal control bar** (per D5/A): on any pointerdown event over the canvas, a thin translucent bar slides up from the bottom (~80px tall, `bg-black/40 backdrop-blur-md`). Contents: timer (left, `text-white/80`, Inter `text-sm`), `[End]` button (right, ghost style on the dark backdrop), `[Pause]` button (center-right). Hides after 3 seconds of no input. Animations: `duration-200` slide-up, `duration-300` fade-out.

**Reduced-motion fallback (per D11/A):** when `window.matchMedia('(prefers-reduced-motion: reduce)').matches`, the canvas renderer enters static mode:
- Both rays visible simultaneously (no alternation), both at 60% opacity.
- No fish drift (silhouettes static).
- No caustic particle motion (static gradient with no particles).
- The submerge transition (D8/A) collapses to a single 300ms fade.
- Audio bilateral runs unchanged — the somatic work happens through hearing.

**Submerge transition (per D8/A):** 2.5s descent before the session starts. Implemented as a full-bleed overlay component:
1. T=0: check-in screen at full opacity.
2. T=0–600ms: page background dims to `rgba(243, 240, 234, 0.6)`; a deep blue `#0a3a5a` gradient slides up from the bottom of the viewport, covering 0→100% from the bottom edge.
3. T=600–1500ms: water sound (slow low-frequency hum) fades in from 0→full volume.
4. T=1500–2500ms: the underwater scene's `<canvas>` mounts and rays begin alternating (audio engine `start()` called inside this window).
5. T=2500ms: transition done; session is live.

Mirror at check-out as **resurface** (2.5s reverse): rays fade out, gradient slides back down, ambient hum fades, scene transitions to the check-out activation-delta visualization (the depth gauge appearing as a diver descending image from 7→4).

Performance budget: 60fps on a 5-year-old laptop. No WebGL yet — keep it canvas. Future Three.js renderer is `Underwater3D.tsx` and slots in behind the same engine.

### Phase 5 — Journal handoff (the punchline of the loop)

There is no journal Zustand store in MoodHaven (verified). The handoff hooks into the existing template-insertion mechanism in `src/pages/WritingView.tsx`: it tracks a `pendingInsertHtml: string | null` state (`WritingView.tsx:179`) and feeds it into the `<RichTextEditor insertHtml={pendingInsertHtml} />` prop (`:918, :1385`). The editor consumes the HTML on mount and inserts it at cursor.

**Session-id encoded in HTML (decided D2, eliminates race condition).** The session_id rides inside the prefilled HTML as a hidden data marker. The journal create flow extracts and strips it at save time. This eliminates the "wrong entry attaches to recent session" race that would otherwise plague the loop: if the user discards the prefill block, the link is gone; if they open journal directly and write an unrelated entry, no marker exists, no link.

The handoff is an inter-view function call:

```typescript
// src/modules/stillhaven/handoff.ts
export async function handoffToJournal(args: {
  setCurrentView: (v: ViewType) => void;
  setPendingInsertHtml: (html: string) => void;
  session: StillSession;
  preSample: ActivationSample;
  postSample: ActivationSample;
}) {
  const html = renderSessionTemplate(args.session, args.preSample, args.postSample);
  args.setPendingInsertHtml(html);
  args.setCurrentView('writing');
}
```

The prefilled HTML carries the session_id in a hidden marker tag at its top:

```html
<span data-still-session-id="018f...uuid" style="display:none"></span>
<h3>Still session — 5 minutes, underwater, audio bilateral</h3>
<p><strong>Activation:</strong> 7 → 4 (down 3)</p>
<p><strong>HRV (oura):</strong> 38 ms → 47 ms</p>
<p><strong>Protocol:</strong> general activation</p>
<p>What came up:</p>
<p></p>
```

**MoodHaven changes required for handoff:**

1. **Lift `pendingInsertHtml` and its setter into `App.tsx`** (or a thin `LayoutContext`). ~20-line refactor — same value flows into `<RichTextEditor insertHtml=...>` as before. No behavior change for existing template usage.
2. **One new method on `journalService`:** `linkEntryToSession(entryId: string, sessionId: string) → Promise<void>`. Implementation calls a new Rust command `link_journal_entry_to_session(entry_id, session_id)` that does a single SQLite UPDATE on `journal_entries.session_id`. Browser-invoke shim mirrors with an IndexedDB update. ~40 lines across 4 files (journalService.ts, browser-invoke.ts, commands/journal.rs, db layer).
3. **`journalService.createEntry()` becomes session-aware:** before save, parse the incoming HTML for `<span data-still-session-id="...">`. If present, capture the id, REMOVE the span from the saved content, and after the entry creation succeeds, call `linkEntryToSession(newEntryId, capturedSessionId)`. Wrapped in try/catch — if linking fails, log a warning but don't fail the entry save.

Tags `still`, `bls`, and the protocol name are appended to the entry's tag list at save time via the existing tag system (also driven by parsing the prefilled HTML's data attributes).

### Phase 6 — Wire pattern tracking (1 session)

Add `'stillSessions'` to the ViewType union (deferred from Phase 0 because the data didn't exist yet). The new view queries `still_list_sessions` + activation samples and renders:

- Activation delta over last 30 days (line chart, before vs after, average gap)
- Protocol frequency (bar chart)
- Time-of-day pattern (when does activation happen most)

**Charting:** reuse the existing MoodHaven analytics components in `src/components/analytics/` (e.g., `JournalingHabits.tsx`'s chart pattern). If they don't fit, defer adding a new chart library to a follow-up phase — for MVP, a simple HTML/SVG bar+line render is enough. **No new dependency** (no Recharts, no Chart.js) unless explicitly approved later.

**Closed-loop UX:** the `<ProtocolPicker />` in Phase 4 reads the last 7 days of session data and surfaces a "you've used `fake_danger` 4 times this week — average activation drop 2.5 points" hint. This is what closes the data loop and earns the "data path is the product" claim.

## Open Questions

0. **Lift `pendingInsertHtml` into App.tsx vs add a LayoutContext.** Both work; LayoutContext is cleaner if more cross-view handoffs are coming, App.tsx prop-drilling is less code today. Pick at implementation time based on current App.tsx complexity.

1. **Web Vibration API for haptic bilateral on phones:** when targeting Cardboard / phone-VR later, does Web Vibration give precise enough timing? May need to fall back to silent audio "ticks" the phone speaker plays into the headset.
2. **Encryption of session metadata.** Journal text is AES-256-GCM. Should activation ratings + protocol names be encrypted too? Probably not — they're not sensitive in isolation, and indexing/charting becomes painful if they are. **Tentative call:** session/sample tables stored in plaintext (like `mood_daily_stats`); the *journal entry* derived from a session remains encrypted as today.
3. **Spatial audio for the underwater scene.** `PannerNode` with HRTF panning vs. simple stereo. HRTF is expensive on low-end devices. Defer to post-MVP — `StereoPannerNode` is plenty for the bilateral cue itself.
4. **Mobile / Cardboard timing.** WebXR is in the renderer abstraction from day one, but no actual WebXR scene ships in MVP. Confirm later: does the underwater 2D scene work in a Quest browser as a stepping stone, or do we go straight to a 3D port?
5. **Rename `journal.moodhaven.app` to `app.moodhaven.app`?** Subjective branding call. Functionally optional — `journal.moodhaven.app/still` works. Worth a separate /design-consultation discussion.

## Success Criteria

- One uninterrupted run of the core loop (notice → quick-launch shortcut → check in → 5-min audio session in the underwater scene → check out → land in a pre-filled journal entry that saves successfully) on the Tauri desktop build.
- The session shows up in `still_list_sessions` with both activation samples and a linked `journal_entries.session_id`.
- An abandoned session (modal closed mid-flight) is correctly marked `abandoned_at` and surfaces a resume/discard prompt on next visit.
- An attempt to enter `/still` while the app is locked routes to the unlock prompt before any check-in UI renders.
- `npm run check:all` (typecheck + lint + test + cargo deny + cargo audit) passes.
- The `no-clinical-language` lint rule fails CI if "EMDR" is added to any TS source file (`src/**`).
- The Rust build.rs lint fails `cargo check` if "EMDR" is added to any Rust source file (`src-tauri/src/**`).
- Web build at `journal.moodhaven.app/still` works equivalently: IndexedDB persistence, no Tauri-only commands invoked, browser-invoke shim covers every `still_*` command. Tested in CI via the existing `npm run build:web` flow.
