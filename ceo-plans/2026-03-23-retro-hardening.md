## Work Stream 1: Bug Fixes

### A-04 — MediaStream / AudioContext cleanup on unmount

**File:** `src/hooks/useAudioRecorder.ts`

**Problem:** The `cleanup()` function (line 121) properly stops all tracks, disconnects the processor,
and closes the AudioContext. But there is NO `useEffect(() => () => cleanup(), [cleanup])` guard.
If the component using this hook unmounts mid-recording (e.g. user navigates away), the OS mic
indicator stays on indefinitely.

**Fix:**
Add a `useEffect` at the bottom of `useAudioRecorder` that calls `cleanup` on unmount:

```typescript
// Cleanup on unmount — prevents mic-indicator leak
useEffect(() => {
  return () => {
    cleanup();
  };
}, [cleanup]);
```

**Effort:** ~5 LOC. No new deps. Test: add a test that simulates unmount during recording.

### A-05 — `cancel()` doesn't abort in-flight `transcribeAudio` / `formatTranscript`

**File:** `src/hooks/useSpeechToText.ts`

**Problem:** `cancel()` (line 194) resets state flags but the `stopAndTranscribe` async chain continues
running. If `transcribeAudio` invoke or `formatTranscript` fetch complete after cancellation,
they still call `setFormattedResult` — surfacing a result the user explicitly dismissed.

**Current cancel():**
```typescript
const cancel = useCallback(() => {
  cancelRecording();
  setIsTranscribing(false);
  setIsFormatting(false);
  setTranscribeError(null);
}, [cancelRecording]);
```

**Fix:**
Add a `cancelledRef = useRef(false)` to `useSpeechToText`. Set it `true` in `cancel()`.
In `stopAndTranscribe`, check `if (cancelledRef.current) return null` after each `await`.
Reset it to `false` at the start of `stopAndTranscribe`.

`formatTranscript` in `aiService.ts` already has internal AbortControllers for its own timeouts.
We do NOT need to thread a signal into `formatTranscript` — checking `cancelledRef` after
the `await formatTranscript(...)` call returns is sufficient to prevent stale setState.

```typescript
const cancelledRef = useRef(false);

const stopAndTranscribe = useCallback(async () => {
  cancelledRef.current = false;
  const audioBuffer = await stopRecording();
  if (!audioBuffer || cancelledRef.current) return null;

  setIsTranscribing(true);
  let rawText: string;
  try {
    const result = await transcribeAudio(audioBuffer, settings.model);
    rawText = result.text;
  } catch (err) { ... }

  if (cancelledRef.current) return null; // ← check after transcribe

  // ... formatting ...
  if (cancelledRef.current) return null; // ← check after format

  setFormattedResult({ ... });
  return null;
}, [...]);

const cancel = useCallback(() => {
  cancelledRef.current = true;  // ← signal
  cancelRecording();
  setIsTranscribing(false);
  setIsFormatting(false);
  setTranscribeError(null);
}, [cancelRecording]);
```

**Effort:** ~15 LOC change in useSpeechToText.ts. No new deps.

### A-08 — TipTap `insertContent` HTML injection via L2/L3 formatter response

**File:** `src/components/editor/RichTextEditor.tsx`

**Problem:** Three `insertContent` calls use external strings as HTML:
- Line 313: `insertContent(text)` — where `text` is `stopAndTranscribe()` return (L1 or L2/L3 fallback)
- Line 325: `insertContent(formattedResult.formatted)` — L2/L3 Ollama or OpenAI response
- Line 333: same (insert-at-cursor variant)

Ollama or OpenAI could return `<a href="javascript:...">`, `<script>`, or similar.

**Fix options:**
- **Option A: DOMPurify** — add `dompurify` npm dep + `@types/dompurify`, sanitize before every `insertContent`. Keeps rich formatting if LLM returns markdown-converted HTML.
- **Option B: `insertText` for LLM responses** — use `insertText` (plain text only) for L2/L3 output. Loses any HTML formatting but LLM transcripts are plain text anyway.
- **Option C: Escape HTML then insertContent** — convert `<` → `&lt;` before insert. Safe but destroys any legitimate paragraph HTML.

**Recommendation: Option B** — LLM transcript responses are plain text with newlines, not HTML. Using `insertText` for L2/L3 output is correct semantically AND eliminates the injection surface. For L1 (our own `cleanTranscript` output), `insertContent` is safe since the output is generated by our regex rules.

**Implementation:**
1. In `RichTextEditor.tsx`, split insertContent calls by source:
   - L1 `stopAndTranscribe` return (line 313): keep `insertContent` — it's our code output
   - `formattedResult.formatted` where `source === 'ollama' | 'openai'` (lines 325, 333): use `insertText` instead
   - `formattedResult.raw` (line 339): use `insertText` — always whisper raw output

**Effort:** ~10 LOC change. No new deps.

## Work Stream 1 (addition): A-10 — `isAvailable` wired to stale settings flag

**File:** `src/hooks/useSpeechToText.ts` — line 213

**Problem:** `isAvailable: settings.enabled && settings.modelDownloaded` uses a stale settings flag.
`checkAvailability()` already hits the filesystem via `checkModelStatus()`. But the returned
`isAvailable` from the hook uses the settings flag, not the checked result.

**Fix:** Wire `availabilityResultRef.current` into the returned `isAvailable`:
```typescript
isAvailable: settings.enabled && availabilityResultRef.current,
```
This means `isAvailable` starts `false` until the first `checkAvailability()` completes — which
is called in `startRecording()`. The mic button will be disabled until the user tries to record once
and the filesystem check passes. This is acceptable UX.

**Note:** If we want `isAvailable` to update proactively on mount, we'd call `checkAvailability()` in
a `useEffect`. Defer that to A-10 proper. For now, just stop lying about state from stale settings.

**Effort:** 1 LOC change. Zero risk.

- **useEffect cleanup dep `[cleanup]`**: `cleanup` is `useCallback` with `[]` deps — stable reference. The effect runs once, returns once. No issues.

- **TipTap `insertText` vs `insertContent`:** Confirmed `insertText` exists in `@tiptap/core` v3 bundle (checked node_modules). `insertText(text)` in TipTap maps to ProseMirror's `insertText` which is always safe.
