## Concerns / blockers

- **Pre-v1.8.0 backups are plaintext at the DB level** — SQLCipher encryption was inert from v1.7.0 through v1.8.0; old on-disk copies (backups, disk images, cloud WebDAV copies) were never DB-encrypted and remain so. Re-encryption applies going forward only. Users on v1.7.x should be prompted to upgrade, but there is currently no explicit in-app notice surfaced to upgrading users.
- **Hotfix chain on release day** — v1.8.1 and v1.8.2 both shipped the same day as v1.8.0, patching a broken updater signing pipeline and incomplete OsRng seeding. The rapid hotfix cadence suggests the CI release pipeline and OAuth hardening paths need more pre-release coverage before major version bumps.
- **Semgrep cloud ruleset unreachable — 8+ consecutive weeks** — `semgrep.dev` returns HTTP 403; cloud rulesets `p/security-audit`, `p/typescript`, `p/owasp-top-ten` are all unavailable. Local rules only. Issue #95 notes this; no fix has landed. `SEMGREP_APP_TOKEN` or a self-hosted rules mirror should be prioritized.

Some have been outstanding 4–7 weeks. These are all confirmed false positives with recommended suppressions spelled out in the issues — they just need to be applied.
- **High PT patch velocity** — PT4 through PT10 resolved in ~5 days is impressive but reflects a significant pre-existing attack surface. Once the Android Phase 2 PRs land, a deliberate slower-cadence security review pass before v1.9.0 would reduce the risk of another rapid hotfix chain.
